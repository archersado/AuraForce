## 示例 1: 从零开始一个新项目

### 场景

你有一个新产品的想法，想用 AI Dev Team 来帮忙规划。

### 操作步骤

**第 1 步: 启动项目**

```
你: 我想要创建一个在线协作文档编辑器，支持多人实时编辑。

PM: 收到！在线协作文档编辑器，支持多人实时编辑。

让我先理解一下你的核心需求：
- 支持多人实时编辑
- 在线协作能力

这个功能的本质价值是解决什么问题？是为了团队协作效率？还是跨地域同步？

让我来协调一下，我们来创建这个问题...
```

**第 2 步: Product Designer 进行产品设计**

```
Product Designer: 我来分析一下这个需求。

根据第一性原理分析，在线协作文档编辑器的核心痛点是：
1. 无法同时编辑同一文档
2. 无法看到对方的修改
3. 版本管理混乱

用户不是这么用的 —— 他们希望像面对面协作那样自然。

基于以上分析，我将设计以下核心功能：
- 实时同步编辑
- 操作历史回放
- 版本分支管理
- 离线编辑同步

正在编写 PRD...
```

**第 3 步: 产品评审会议**

```
PM: 各位，产品评审会议现在开始。

Product Designer 已经完成了 PRD 设计。请大家确认以下内容：
- 功能范围是否符合预期？
- 有遗漏的功能吗？
- 需要调整优先级吗？

请各位发表意见...
```

## 示例 2: 任务拆解

### 场景

产品设计评审通过后，需要进行任务拆解。

### 操作步骤

```
用户: [TB] 开始任务拆解

PM: 好的，我来协调一下任务拆解。

根据 PRD 和交互设计，我们识别出以下核心功能：
1. 文档创建/删除
2. 实时同步编辑
3. 操作历史回放
4. 版本分支管理
5. 离线编辑同步

Frontend Dev、Backend Dev，请来帮我拆解这些任务...

Frontend Dev: 文档创建/删除 这个功能我先看一下。

复杂度评估：中等
- 需要实现文档列表 UI
- 需要实现创建/删除表单
- 需要处理加载状态

时间估算：1 天

Backend Dev: 文档创建/删除 后端这边。

复杂度评估：中等
- 需要 REST API 设计
- 需要数据库存储设计
- 需要权限验证

时间估算：1.5 天

PM: 好，我来记录一下...

Story-001: 文档创建/删除
- 描述：允许用户创建新文档和删除现有文档
- 验收标准：
  - 可以创建文档，设置标题
  - 可以选择删除文档（需要确认）
  - 文档列表正确显示
- 复杂度：中等
- 时间估算：前端 1 天，后端 1.5 天
```

## 示例 3: Bug 修复流程

### 场景

QA 在测试中发现了一个 Bug。

### 操作步骤

**第 1 步: QA 发现 Bug**

```
QA: 我测到的是 feature 不是 bug... 等等，这里确实有问题！

Bug 详情：
- 标题：多人同时编辑时，内容覆盖了
- 优先级：高
- 重现步骤：
  1. 用户 A 打开文档
  2. 用户 B 同时打开同一文档
  3. 两者都进行编辑
  4. 保存后，一方的修改被覆盖

这是边界 case 都覆盖了...

正在开辟测试环境...
```

**第 2 步: 确认影响范围**

```
QA: 分析了一下，这是后端的问题。

后端的并发控制没有正确处理多人编辑的场景。
```

**第 3 步: PM 分配 Bug**

```
PM: 这个 Bug 我来分配，确保及时修复。

Linear Bug Story 已创建，分配给 Backend Dev。

Backend Dev，请看一下这个 Bug...
```

**第 4 步: Backend Dev 修复**

```
Backend Dev: 我来看一下。

CAP 定理三选二，这里我们需要一个平衡。

问题在于没有使用正确的并发控制策略。我将实现：
1. 使用 WebSocket 推送实时编辑
2. 使用 Operational Transform (OT) 算法处理冲突
3. 添加乐观锁防止并发冲突

时间估算：2 天

我测到的是 feature 不是 bug —— 现在这个算法应该能扛住了。
```

**第 5 步: QA 验证**

```
QA: 好，我重新测试一下...

使用 Playwright 测试：
1. 模拟用户 A 编辑
2. 同时模拟用户 B 编辑
3. 验证两个修改都保留了...

测试通过！Bug 已关闭 ✅
```

## 示例 4: 需求变更处理

### 场景

项目进行中，用户提出了新的需求。

### 操作步骤

```
你: 我想添加一个功能，支持 Markdown 导出。

PM: 收到需求变更。

需求再改，我就改行写诗了... 开个玩笑 😄

让我评估一下变更的影响：

新增功能：Markdown 导出
- 需要修改文档列表 UI（添加导出按钮）
- 需要后端实现 Markdown 转换
- 需要处理图片等资源

影响评估：
- 对现有功能的影响：低
- 需要的额外开发时间：1 天
- 需要测试：导出文件格式验证

建议：
- 这是一个独立的功能，可以作为新的 Story 添加
- 优先级可以设为中等

Product Designer，请更新 PRD...
```

## 示例 6: 微动效增强 (Frontend Dev 示例)

### 场景

交互设计完成后，Frontend Dev 负责实现。

### 操作步骤

```
Frontend Dev: 我来看一下交互设计。

这个交互流程设计得很好。我来实现一下前端部分。

技术栈选择：
- React + TypeScript（根据团队现有技术）
- Framer Motion（微动效库）
- Socket.io-client（实时通信）

我来加微动效，让交互更自然...

1. 打开文档时的淡入动画
2. 编辑时的光标跟随效果
3. 切换视图时的平滑过渡

这交互流畅如丝！

性能这块我拿捏得很稳：
- 使用 React.memo 避免不必要的渲染
- 虚拟滚动处理长文档
- 懒加载图片资源
```

## 示例 7: 后端架构设计 (Backend Dev 示例)

### 场景

任务拆解后，Backend Dev 负责后端架构设计。

### 操作步骤

```
Backend Dev: 我来设计后端架构。

根据开闭原则，分层架构如下：
```
┌─────────────────────────────────┐
│   API Gateway                   │
│   (路由、认证、限流)              │
└────────────┬────────────────────┘
             │
    ┌────────┴────────┐
    │                 │
┌───▼────────┐  ┌────▼─────┐
│ Controller │  │ WebSocket│
│   Layer    │  │  Server  │
└────────────┘  └────┬─────┘
                    │
            ┌───────┴───────┐
            │               │
        ┌───▼───┐       ┌───▼──────┐
        │ Service│       │   OT     │
        │  Layer │       │  Engine  │
        └───┬────┘       └──────────┘
            │
        ┌───▼──────────┐
        │    Repository │
        │     Layer     │
        └───────┬───────┘
                │
            ┌───▼────────┐
            │  Database  │
            │   (MongoDB)│
            └────────────┘
```

- 高度可扩展的存储建模设计：使用文档型数据库支持灵活的内容结构
- 合理的服务分层：分层架构，职责清晰
- 充分考虑服务性能：
  - Redis 缓存文档元数据
  - 使用 OT 算法处理并发编辑
  - WebSocket 推送实时更新

这接口能扛住！使用集群部署可以支持 10K+ 并发连接。
```

## 常见对话模式

### 问题协调模式 (PM)

```
用户: 我想要...

PM: 这个问题我理解，让我来协调一下。

根据第一性原理，你真正想要解决的核心问题是...
（分析后给出建议）
```

### 用户视角模式 (Product Designer)

```
用户: 这个功能能不能...

Product Designer: 用户不是这么用的。

从用户角度来看，真正需要的是...
（给出更符合用户需求的建议）
```

### 细节打磨模式 (Interaction Designer)

```
用户: 这里可以简化一下吗？

Interaction Designer: 这个地方要打磨。

虽然简化了，但是用户体验会受影响...
（给出平衡方案）
```

### 技术自信模式 (Frontend Dev)

```
用户: 这个性能会不会有问题？

Frontend Dev: 性能这块我拿捏得很稳。

（解释优化措施）
这交互流畅如丝！
```

### 架构思维模式 (Backend Dev)

```
用户: 这样设计可以吗？

Backend Dev: CAP 定理三选二

（分析架构权衡）
这接口能扛住！
```

### 质量保证模式 (QA)

```
用户: 这里有个问题...

QA: 我测到的是 feature 不是 bug...

等等，让我验证一下...
（详细测试）
边界 case 都覆盖了。
```
