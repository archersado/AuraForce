# ⚠️ 重要更新：设计优先开发流程

**更新时间：** 2025-02-02
**影响版本：** 所有新的需求

---

## 🔄 流程变更

### 旧流程（Sprint 1）❌
```
PM 分配任务
    ↓
产品、开发、测试并行开始
    ↓
各自完成交付
```

### 新流程（Sprint 2 及以后）✅
```
PM 分配需求给 Product Designer
    ↓
【设计阶段】
Product Designer 完成 PRD + UI/UX 设计 (2-3天)
    ↓
【设计评审】全员参与评审 (30-45分钟)
    ↓
【开发阶段】（设计评审通过后）
Frontend Lead 基于 UI/UX 设计开发
Backend Engineer 基于 PRD 开发
    ↓
【测试阶段】
QA Engineer 基于设计和需求测试
```

---

## ⚠️ 关键要求

### 1. 设计必须优先 ⭐⭐⭐
- ❌ 设计未完成前，**不能**开始开发
- ❌ 设计未评审通过，**不能**进入开发阶段
- ✅ 必须等待 Product Designer 完成所有设计文档

### 2. 设计必须评审 ⭐⭐⭐
- ✅ 全员参与：PM + Product Designer + Frontend Lead + Backend Engineer
- ✅ 评审内容：PRD 评审、UI/UX 评审、技术可行性评估
- ✅ 评审结果：通过 → 进入开发；修改 → 重新评审

### 3. 开发基于设计 ⭐⭐⭐
- ✅ Frontend Lead 基于**UI/UX 设计**开发
- ✅ Backend Engineer 基于**PRD**开发
- ❌ 不得在没有设计文档的情况下开发

### 4. 测试基于设计和需求 ⭐⭐
- ✅ QA Engineer 基于**PRD**和**设计稿**测试
- ✅ 验证功能符合 PRD
- ✅ 验证 UI/UX 符合设计稿

---

## 📋 设计交付标准

Product Designer 完成时必须交付：

1. ✅ **完整的 PRD 文档**
   - 产品概述
   - 目标用户
   - 功能需求详细定义
   - 非功能需求（性能、安全等）

2. ✅ **完整的 UI/UX 设计**
   - 用户流程图
   - 交互设计
   - 设计规范（颜色、字体、尺寸）
   - 可交付的设计稿（Figma 或详细描述）

3. ✅ **功能规格说明**
   - 功能规格
   - 交互规格
   - 数据规格

## 🗓️ 时间分配建议

| 阶段 | 时间 | 负责人 | 参与者 |
|------|------|--------|--------|
| PM 需求分析 | 1-2 小时 | PM | - |
| PM 分配需求 | 30 分钟 | PM | - |
| Product Designer 设计 | 2-3 天 | Product Designer | - |
| **设计评审** | **30-45 分钟** | **PM** | **全员** ⭐ |
| Frontend 开发 | 2-3 天 | Frontend Lead | - |
| Backend 开发 | 2-3 天 | Backend Engineer | - |
| 集成联调 | 1 天 | Frontend + Backend | - |
| QA 测试 | 2-3 天 | QA Engineer | - |
| Sprint Review | 1 小时 | PM | 全员 |

## ✅ 检查清单

### 设计完成后，PM 检查：
- [ ] PRD 文档完整
- [ ] UI/UX 设计文档完整
- [ ] 交互流程清晰
- [ ] 设计规范明确
- [ ] 可交付的设计稿存在

### 设计评审阶段检查：
- [ ] 全员参与评审
- [ ] PRD 评审通过
- [ ] UI/UX 评审通过
- [ ] 技术可行性确认
- [ ] 风险已识别

### 开发开始前检查：
- [ ] 设计已评审通过
- [ ] 前端有 UI/UX 设计稿
- [ ] 后端有 PRD 和规格说明
- [ ] QA 有设计和需求文档用于测试
