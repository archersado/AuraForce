## 概述

### 什么是 AuraForce 研发项目管理？

AuraForce 研发项目管理是一套标准化的项目管理体系，旨在：

- ✅ 规范化需求到归档的完整生命周期
- ✅ 分离项目管理、产品设计、技术开发职责
- ✅ 提供标准化的文档模板和工作流程
- ✅ 实现可追踪、可复用、可传承的项目管理

### 核心理念

**需求 🠲 设计 🠲 开发 🠲 测试 🠲 归档**

完整覆盖从用户需求到项目归档的全流程，确保每个环节都有明确的责任人和交付物。

## 核心原则 ⭐

### 1. 设计优先开发 ⭐⭐⭐

**必须遵循的流程：**
```
需求分析 → 产品设计 → 设计评审 → 开发实施 → 测试验收
```

**关键要求：**
- ❌ 设计未完成前，**不能**开始开发和测试
- ✅ Product Designer 必须先完成 PRD、UI/UX 设计、交互设计
- ✅ 开发团队必须基于设计稿和 PRD 进行开发
- ✅ QA Engineer 必须基于设计和需求文档进行测试

**为什么重要：**
- 避免前后端设计和代码不一致
- 减少重复修改和沟通成本
- 确保产品符合设计和交互定义
- 提高开发效率和质量

### 2. 文档驱动开发

所有工作必须有文档支撑：
- 需求有 REQ 文档
- 设计有 PRD 和 UI/UX 文档
- 开发有技术文档和 API 文档
- 测试有测试计划和测试报告

### 3. 角色职责分离

- **PM** - 项目管理和协调
- **Product Designer** - 产品和交互设计
- **Development Team** - 技术实现
- **QA Engineer** - 质量保证

### 4. 持续追踪和反馈

- 每日进度更新
- 定期风险识别
- Sprint 回顾和改进

## 工作流程

### 完整生命周期

```
用户提出需求
    ↓
PM: 接收并确认需求 (10-15分钟)
    ↓
PM: 创建需求 Story REQ-XXX (10-20分钟)
    ↓
PM: 拆解 Epic → Story → Task (20-30分钟)
    ↓
PM: 指派需求给 Product Designer
    ↓
【设计阶段】⭐
Product Designer: 需求分析 (15-30分钟)
    ↓
Product Designer: 编写 PRD (30-60分钟)
    ↓
Product Designer: 创建 UI/UX 设计 (30-60分钟)
    ↓
Product Designer: 编写功能规格说明 (20-30分钟)
    ↓
PM: 组织设计评审 (30-45分钟) ⭐ 新增
    ↓
【开发阶段】⭐
PM: 分配开发任务给 Frontend Lead 和 Backend Engineer
    ↓
Frontend Lead: 基于 UI/UX 设计开发 (2-3天)
    ↓
Backend Engineer: 基于 PRD 开发 API (2-3天)
    ↓
Frontend + Backend: 集成和联调 (1天)
    ↓
【测试阶段】⭐
QA Engineer: 基于设计和需求测试 ⭐ (2-3天)
    ↓
【验收阶段】⭐
PM: 验收和 Sprint Review
    ↓
PM: 项目归档
```

### 详细步骤

#### 步骤 1: 提出需求

**用户 → PM**

```
sessions_send("PM session key", "我想添加一个新功能：...")
```

**需求内容应包含：**
- 功能描述
- 用户场景
- 预期效果
- 优先级（可选）

---

#### 步骤 2: PM 处理需求

PM 会执行以下操作：

1. **需求分析** (10-15 分钟)
   - 理解用户需求
   - 评估可行性
   - 明确范围

2. **创建需求文档** (10-20 分钟)
   - 在 `docs/pm/requirements/` 创建 `REQ-XXX-[标题].md`
   - 使用需求 Story 模板
   - 更新 requirements/README.md

3. **任务拆解** (20-30 分钟)
   - 创建 Epic（大功能）
   - 拆解 Story（功能模块）
   - 创建 Task（具体开发任务）

4. **分配给 Product Designer** (5-10 分钟) ⭐ 关键
   - 发送需求给 Product Designer
   - 等待 PRD 和 UI/UX 设计完成
   - ⚠️ 设计完成前不分配开发任务

---

#### 步骤 3: 产品设计 ⭐ 必须

**PM → Product Designer**

⚠️ **重要：** 设计阶段必须完成，才能进入开发阶段

Product Designer 执行以下操作：

1. **需求分析** (15-30 分钟)
   - 理解 PM 分配的需求
   - 分析用户场景
   - 明确产品目标

2. **编写 PRD** (30-60 分钟)
   - 在 `docs/product/prd/` 创建 `PRD-XXX-[产品名].md`
   - 包含产品概述、目标用户、功能需求等
   - 为开发团队提供清晰的产品定义

3. **UI/UX 设计** (30-60 分钟) ⭐ 重要
   - 在 `docs/product/design/` 创建设计文档
   - 用户流程、交互设计
   - 设计规范（颜色、字体、尺寸）
   - 组件设计
   - **必须有可交付的设计稿**（Figma、图片或详细描述）

4. **编写规格说明** (20-30 分钟)
   - 在 `docs/product/specs/` 创建规格文档
   - 功能规格、交互规格
   - 数据规格

5. **向 PM 汇报设计完成** (5-10 分钟) ⭐

**设计完成标准：**
- ✅ PRD 文档完整
- ✅ UI/UX 设计文档完整
- ✅ 交互流程清晰
- ✅ 设计规范明确
- ✅ 开发团队可以据此开始工作

---

#### 步骤 3.5: 设计评审 ⭐ 新增 ⭐⭐⭐

**PM 组织，全员参与**

**参与者：**
- PM
- Product Designer
- Frontend Lead
- Backend Engineer
- （可选）Database Architect

**评审内容：**
1. **PRD 评审** (10-15 分钟)
   - 产品目标明确
   - 功能需求清晰
   - 技术可行性评估

2. **UI/UX 评审** (10-15 分钟)
   - 设计是否符合需求
   - 交互流程合理
   - 技术实现可行性
   - 估算工作量

3. **问题识别和记录** (5-10 分钟)
   - 识别潜在问题
   - 记录风险
   - 确认设计修改（如需要）

4. **评审结论** (5 分钟)
   - ✅ 通过 - 进入开发阶段
   - ⏸️ 修改 - Product Designer 修改后重新评审
   - ❌ 挂起 - 需求不明确，回到 PM

**评审输出：**
- `docs/pm/tracking/design-review-XXX.md` - 设计评审报告

**评审结论通过后，分配开发任务。**

---

#### 步骤 4: 技术开发 ⭐ 设计完成后开始

**PM → Development Team**

⚠️ **重要：** 开发必须基于 PRD 和 UI/UX 设计

**Frontend Lead 执行：**

1. **基于 UI/UX 设计开发** (2-3 天)
   - 阅读设计文档和设计稿
   - 实现 UI 组件
   - 集成交互逻辑
   - 参考设计规范（颜色、字体、尺寸）

2. **编写单元测试** (半天)

**Backend Engineer 执行：**

1. **基于 PRD 开发 API** (2-3 天)
   - 阅读功能规格和需求
   - 设计 API 接口
   - 实现后端逻辑
   - 编写 API 文档

2. **编写单元测试** (半天)

**联调阶段：**
- 前后端集成测试
- 接口联调
- Bug 修复

---

#### 步骤 5: 测试和质量保证 ⭐ 基于设计和需求

**PM → QA Engineer**

⚠️ **重要：** 测试必须基于 PRD 和设计文档

QA Engineer 执行：

1. **基于 PRD 的功能测试**
   - 对照 PRD 的功能需求
   - 验证所有功能点

2. **基于设计的 UI/UX 测试**
   - 对照设计稿验证 UI
   - 验证交互流程
   - 验证设计规范（颜色、字体、尺寸）

3. **集成测试**
   - 前后端集成测试
   - 跨浏览器测试

4. **编写测试报告** (半天-1天)

---

#### 步骤 6: 项目追踪和验收

**PM 持续追踪：**

- 创建 Sprint 追踪表
- 更新里程碑状态
- 管理项目风险
- 定期生成进度报告

**验收阶段：**
- PM 验收
- 用户验收
- Sprint Review

---

#### 步骤 7: 项目归档

**PM 执行：**

- 更新 Epic、Story、Task 为完成状态
- 创建归档文档
- 移动到 `pm/archived/`
- 撰写项目复盘

## 常见问题

### Q1: 为什么设计必须优先？

**A:** 因为：
- 避免前后端设计和代码不一致
- 减少重复修改和沟通成本
- 确保产品符合设计和交互定义
- 提高开发效率和质量

### Q2: 设计评审必须全员参与吗？

**A:** 是的，强烈建议。因为：
- Frontend Lead 需要确认 UI 技术可行性
- Backend Engineer 需要确认交互技术可行性
- PM 需要确认设计符合需求
- 提前识别问题，避免后期返工

### Q3: 如果设计评审不通过怎么办？

**A:**
1. 记录问题和修改建议
2. Product Designer 修改设计
3. 重新组织评审
4. 直到评审通过，进入开发阶段

### Q4: 开发可以在设计评审前开始吗？

**A:** 不可以。⚠️
- 严格的流程：设计 → 评审 → 开发
- 如果需要提前做技术预研，必须明确是"预研"，不是正式开发
- 正式开发必须基于通过评审的设计

### Q5: PM 如何知道设计已完成？

**A:**
1. Product Designer 主动汇报 "设计完成"
2. PM 查看交付的文档：PRD + UI/UX 设计 + 规格说明
3. 验证设计完整性
4. 组织设计评审

### Q6: Sprint 1 的流程有问题吗？

**A:** 是的，有问题。⚠️
- Sprint 1 采用了并行模式，产品和开发同时开始
- 这不符合"设计优先开发"原则
- 从 Sprint 2 开始，严格按照"设计优先"流程执行
